# Story: Update /create-missing-stories Command Documentation with Story Quality Requirements

## Description

**As a** DevForgeAI framework user running the `/create-missing-stories` command,
**I want** clear documentation explaining that generated stories must have verified_violations sections,
**so that** I understand the story quality requirements upfront and know why stories might not be created.

**Context from RCA-020:**
The `/create-missing-stories` command documentation doesn't explain that generated stories must include verified_violations sections with specific file paths and line numbers. Users running the command may be surprised when story creation fails due to evidence verification requirements.

**Documentation Gap:**
Current documentation explains:
- How to run the command
- What arguments it accepts
- What output to expect

Current documentation does NOT explain:
- Story quality gates (verified_violations requirement)
- Why stories might not be created (claim verification failures)
- What "verified claims" means in practice

**User Experience Improvement:**
By documenting these requirements upfront:
1. Users understand what "quality stories" means
2. Users aren't surprised by verification failures
3. Users can pre-validate their epic claims before running the command
4. The framework's commitment to evidence-based specifications is clear

## Acceptance Criteria

### AC#2: Required Story Elements Documented

**Given** the Story Quality Gates section
**When** a user reads it
**Then** they understand that all generated stories must include:

1. **`verified_violations` section in technical specification** (if claims involve modifications)
2. **Specific file paths and line numbers** (not generic ranges)
3. **Verification that target files exist** and contain claimed violations
4. **No placeholder values** (TBD, TODO, etc.)

**Documentation Format:**
```markdown
**Required Story Elements:**

All stories generated by `/create-missing-stories` must include:

1. **verified_violations section** - When technical specifications claim violations exist (e.g., "Remove Bash commands"), the story MUST include a verified_violations section proving those violations were found.

2. **Specific file paths and line numbers** - Verified violations must include actual line numbers like `lines: [469, 598, 599]`, not generic descriptions like "around line 500".

3. **Target file validation** - All files mentioned in verified_violations are checked to exist and contain the claimed violations during story creation.

4. **No placeholders** - Values like "TBD", "TODO", or "N/A" are not permitted in verified_violations sections.
```

---

### AC#3: Story Creation Failure Explanation

**Given** a user whose story creation failed due to verification issues
**When** they read the documentation
**Then** they understand:
1. WHY the story was not created
2. WHAT claims could not be verified
3. HOW to fix the issue (check file paths, verify claims manually)

**Documentation Format:**
```markdown
**Why Stories Might Not Be Created:**

If a story is not created, the command will display an error message indicating which claim could not be verified.

Common reasons for story creation failure:

| Failure Reason | Cause | Fix |
|----------------|-------|-----|
| Target file not found | File path in epic is incorrect or outdated | Update epic with correct file path |
| Claim not verified | File doesn't contain the claimed violation | Update epic description to match reality OR verify file contains expected content |
| Generic description | Epic uses vague claims without specifics | Add specific counts and locations to epic description |
| Placeholder values | Epic contains TBD/TODO markers | Replace placeholders with actual values |

**Example Error:**
```
❌ CRITICAL: Cannot verify claim: "Remove 5 Bash mkdir commands"
File checked: .claude/commands/ideate.md
Found: 0 matches for "Bash mkdir"
Expected: 5 violations

Fix: Update epic to reflect actual violations, or check file path is correct.
```
```

### AC#4: Evidence Requirement Rationale Documented

**Given** a user reading the documentation
**When** they encounter the evidence requirements
**Then** they understand WHY evidence-based verification is required:

**Documentation Format:**
```markdown
**Why Evidence-Based Verification?**

DevForgeAI enforces evidence-based story creation to ensure:

1. **No false claims** - Every technical specification is backed by verified evidence
2. **Implementable stories** - Developers can trust that claimed violations actually exist
3. **No post-hoc verification** - Stories are ready to implement without re-checking claims
4. **Framework integrity** - Follows Read-Quote-Cite-Verify protocol from `.claude/rules/core/citation-requirements.md`

This requirement was added after RCA-020 discovered stories being created with unverified claims that required manual enhancement later.
```
