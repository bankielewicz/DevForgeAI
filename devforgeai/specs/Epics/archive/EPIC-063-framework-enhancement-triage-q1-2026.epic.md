#### Feature 1: Fix Implementation Notes Template DoD Header

| Field | Value |
|-------|-------|
| Priority | HIGH |
| Points | 1 |
| Type | refactor |
| Effort | 15 minutes |
| Source Recommendation | REC-STORY370-001 |
| Source Story | STORY-370 (Phase 09 framework-analyst) |

**Description:** Remove the `### Definition of Done Status` subsection header from the implementation-notes-template.md that blocks the pre-commit DoD validator. The `extract_section("Implementation Notes")` function stops at the FIRST `###` header it encounters, making ALL DoD checkbox items under this subsection invisible to the validator, which causes false commit failures.

**Implementation Details:**

Target file: `.claude/skills/devforgeai-development/assets/templates/implementation-notes-template.md`
- File size: 18 lines
- Problem: Line 7 contains `### Definition of Done Status` — a `###` subsection header
- The pre-commit validator's `extract_section()` function (documented in `dod-update-workflow.md` lines 116-119) returns content from `## Implementation Notes` up to the NEXT `##` or `###` header
- Since `### Definition of Done Status` appears BEFORE the DoD items, `extract_section()` stops there and returns only the developer info lines, finding 0 DoD items
- The `### Additional Notes` subsection on line 15 should REMAIN since it comes AFTER DoD items and doesn't interfere

Current file content (all 18 lines):
```markdown
## Implementation Notes

**Developer:** {DEVELOPER_NAME}
**Implemented:** {YYYY-MM-DD}
**Branch:** {BRANCH_NAME}

### Definition of Done Status        ← LINE 7: THIS MUST BE REMOVED

<!-- For each completed DoD item: -->
- [x] {DoD item text} - Completed: {brief evidence of completion}

<!-- For each deferred DoD item: -->
- [ ] {DoD item text} - Deferred: {technical justification} (See: STORY-XXX)

### Additional Notes                  ← LINE 15: KEEP THIS

{Optional: Any implementation notes, decisions made, or context for future developers}
```

Expected result after fix — DoD items flat under `## Implementation Notes`, no `###` before them:
```markdown
## Implementation Notes

**Developer:** {DEVELOPER_NAME}
**Implemented:** {YYYY-MM-DD}
**Branch:** {BRANCH_NAME}

<!-- For each completed DoD item: -->
- [x] {DoD item text} - Completed: {brief evidence of completion}

<!-- For each deferred DoD item: -->
- [ ] {DoD item text} - Deferred: {technical justification} (See: STORY-XXX)

### Additional Notes

{Optional: Any implementation notes, decisions made, or context for future developers}
```

**Acceptance Criteria Guidance:**
1. The `### Definition of Done Status` line is removed from the template
2. DoD checkbox items remain directly under `## Implementation Notes` (no intervening `###` headers)
3. The `### Additional Notes` subsection remains intact after DoD items
4. The template produces output compatible with `extract_section("Implementation Notes")` — all DoD items are extractable

**Key References:**
- DoD update workflow: `.claude/skills/devforgeai-development/references/dod-update-workflow.md` (lines 116-119 explain extract_section behavior, lines 136-145 show the exact failure pattern)

#### Feature 6: Add AC-Level Chunking to Compliance Verifier

| Field | Value |
|-------|-------|
| Priority | MEDIUM |
| Points | 2 |
| Type | refactor |
| Effort | 30 minutes |
| Source Recommendation | REC-STORY368-001 |
| Source Story | STORY-368 (Phase 09 framework-analyst) |

**Description:** Update the ac-compliance-verifier workflow to verify acceptance criteria in chunks of 3 when a story has 5 or more ACs, reducing peak token usage by approximately 40%.

**Implementation Details:**

Target file: `.claude/agents/ac-compliance-verifier.md`
- This subagent uses a "fresh-context technique" — it is invoked as a Task() subagent that reads the story file fresh each time
- Currently it loads the entire story and verifies ALL acceptance criteria in a single pass
- For large stories (5+ ACs), this consumes excessive tokens because each AC verification requires loading: the AC definition, implementation evidence, and test evidence simultaneously

Add chunking logic to the verification workflow:

1. **Count ACs:** Parse the story file to count total acceptance criteria (look for `<acceptance_criteria>` XML blocks)
2. **Decision point:**
   - If AC count < 5: verify all at once (EXISTING behavior, no change)
   - If AC count >= 5: chunk into groups of 3
3. **Chunk processing:** For each chunk of 3 ACs:
   - Re-read the story file fresh (already the verifier's pattern)
   - Verify only the ACs in this chunk
   - Store chunk results
4. **Aggregation:** Combine all chunk results into a single verification report
   - The final report format MUST be identical to the current single-pass format
   - Downstream consumers (QA skill Phase 1, DoD update Phase 07) expect this format

Test scenarios to cover:
- 3 ACs → no chunking (single pass)
- 5 ACs → 2 chunks (3 + 2)
- 6 ACs → 2 chunks (3 + 3)
- 8 ACs → 3 chunks (3 + 3 + 2)

**Acceptance Criteria Guidance:**
1. Stories with <5 ACs are verified in a single pass (existing behavior unchanged)
2. Stories with >=5 ACs are verified in chunks of 3
3. Each chunk re-reads the story file fresh
4. Aggregated report format is identical to single-pass format
5. Downstream consumers (QA skill, DoD update) work without modification
6. Token usage reduced for 5+ AC stories (measurable via phase-state.json token tracking)

**Key References:**

- **Probability:** Low
- **Impact:** Medium
