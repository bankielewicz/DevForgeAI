# RCA-047: Orchestrator Test Modification Phase Violation

**Date:** 2026-03-03
**Reported By:** User
**Affected Component:** Orchestrator (opus), implementing-stories skill, test-folder-protection rule
**Severity:** CRITICAL

---

## Issue Description

During STORY-531 `/dev` workflow, the orchestrator (opus) directly modified all 5 test files created by test-automator subagent during Phase 02 (Red). The test files had a bash arithmetic bug (`((PASSED++))` under `set -e` fails when PASSED=0). Instead of returning to Phase 02 to have test-automator regenerate the tests with the fix, the orchestrator:

1. Asked user permission via AskUserQuestion (per test-folder-protection rule)
2. Used `Write()` to rewrite all 5 test files during Phase 03 (Green)
3. Changed `set -euo pipefail` to `set -uo pipefail` and `((PASSED++))` to `PASSED=$((PASSED+1))`

**Expected:** The orchestrator should never directly modify test files. When tests need modification, the workflow should return to Phase 02 and invoke test-automator to regenerate them.

**Actual:** The orchestrator modified the tests itself, causing all 5 test file checksums to mismatch the red-phase snapshot (`devforgeai/qa/snapshots/STORY-531/red-phase-checksums.json`). QA Phase 1.5 (Diff Regression Detection) correctly flagged this as `CRITICAL: TEST TAMPERING` for 5/5 files.

**Impact:**
- All 5 test file checksums mismatched red-phase snapshot
- QA flagged TEST TAMPERING (5/5 files) — correctly
- Story QA approval was subsequently reverted (see RCA-046)
- Test-folder-protection rule was nominally followed (user asked) but semantically violated (wrong remedy chosen)
- Principle that orchestrator never modifies subagent-created tests was violated

## 5 Whys Analysis

**Issue:** The orchestrator directly used Write() on all 5 test files during Phase 03 instead of returning to Phase 02 for test-automator regeneration.

### Why #1: Why did the orchestrator directly modify all 5 test files during Phase 03?

The test files generated by test-automator in Phase 02 had a bash arithmetic bug: `((PASSED++))` under `set -e` causes immediate exit when PASSED=0 (the arithmetic expression evaluates to 0, which is falsy in bash, triggering the `set -e` exit). The orchestrator discovered this during Phase 03 when running tests to verify GREEN state. The orchestrator chose to fix the tests itself using Write() rather than returning to Phase 02.

**Evidence:** Conversation transcript — orchestrator called `Write(file_path="tests/STORY-531/test_ac1_lean_canvas_generation.sh", ...)` for all 5 files during Phase 03. Changes: `set -euo pipefail` → `set -uo pipefail`, `((PASSED++))` → `PASSED=$((PASSED+1))`, `((FAILED++))` → `FAILED=$((FAILED+1))`.

### Why #2: Why did the orchestrator choose to fix tests itself rather than return to Phase 02?

The orchestrator treated this as a "quick infrastructure bug fix" and framed it to the user as such. The AskUserQuestion invocation per test-folder-protection asked "Can I fix the bash arithmetic bug?" with options "Approve fix" / "Deny." The question did not present the correct workflow remedy: "Return to Phase 02 to have test-automator regenerate the tests." The orchestrator rationalized that user approval via AskUserQuestion was sufficient authorization.

**Evidence:** `.claude/rules/workflow/test-folder-protection.md` lines 61-79 — AskUserQuestion protocol provides only "Approve this modification" and "Deny and HALT." No "Return to Phase 02" option.

### Why #3: Why didn't the test-folder-protection rule require returning to Phase 02?

The test-folder-protection rule was designed as a binary gate: approve or deny. It correctly identifies that Phase 03 test writes require AskUserQuestion, but its remediation protocol only offers direct approval or full halt. The rule does not encode the TDD workflow principle that test modifications should flow through the authorized agent (test-automator) during the authorized phase (Phase 02). The rule prevents *silent* tampering but still permits *approved* tampering.

**Evidence:** `.claude/rules/workflow/test-folder-protection.md` lines 86-92 — Phase-Specific Behavior table shows Phase 03 as "All test writes HALT + AskUserQuestion" with no mention of Phase 02 re-entry as the correct remedy. Lines 96-102 — Rationale section discusses risks but proposes no backward-phase mechanism.

### Why #4: Why doesn't the implementing-stories workflow support backward phase transitions for test defects?

The implementing-stories phase state machine is strictly forward-only. The Phase Orchestration Loop (SKILL.md line 149) iterates `FOR phase_num in range(CURRENT_PHASE, 11)` — forward only. The Phase Resumption section (lines 319-332) only describes resuming from the current phase forward. The `devforgeai-validate phase-check` CLI validates forward transitions (e.g., `--from=02 --to=03`) but has no backward transition support. The workflow-deviation-protocol.md defines deviation types as "Phase Skipping," "Subagent Omission," and "Out-of-Sequence" — no "Phase Regression" type.

**Evidence:**
- SKILL.md line 149: `FOR phase_num in range(CURRENT_PHASE, 11)` — forward only
- SKILL.md lines 319-332: Phase Resumption — "resume from current phase" (forward)
- workflow-deviation-protocol.md lines 9-15: Three deviation types, none is "Phase Regression"
- `devforgeai-validate phase-check`: Only validates forward transitions

### Why #5 (ROOT CAUSE): Why was the TDD workflow designed without backward phase transitions for test defects discovered after Phase 02?

**ROOT CAUSE:** The implementing-stories workflow was designed with the implicit assumption that Phase 02 (Red) produces correct, stable test files. The phase state machine, CLI validation, test-folder-protection rule, and workflow deviation protocol all enforce or assume forward-only progression. When test-automator produces tests with infrastructure defects (bash syntax bugs, not business logic errors), the orchestrator faces a false dilemma: (a) fix the tests directly (test-folder-protection violation + checksum mismatch), or (b) deny and HALT permanently (deadlock). The correct path — re-entering Phase 02 with a directive for test-automator to regenerate fixed tests and update the checksum snapshot — does not exist in the framework. The absence of this backward transition path forces the orchestrator into a violation regardless of which option it chooses.

**Evidence:**
- SKILL.md Phase Orchestration Loop: forward-only iteration
- `devforgeai-validate phase-check`: forward-only validation
- test-folder-protection.md: binary approve/deny with no Phase 02 re-entry option
- workflow-deviation-protocol.md: no "Phase Regression" deviation type
- test-integrity-snapshot.md lines 97-99: Overwrite behavior (BR-005) explicitly supports re-running Phase 02 ("supports re-running Phase 02 without manual cleanup") — proving the framework *anticipated* Phase 02 re-entry but never built the transition mechanism

   ```

```python
def phase_reset_command(
    story_id: str,
    project_root: str,
    target_phase: str,
    format: str = "text",
    workflow: str = "dev"
) -> int:
    """
    Reset phase state to an earlier phase (backward transition).

    Used for Phase Regression when test infrastructure defects require
    returning to Phase 02 for test-automator regeneration.

    Args:
        story_id: Story identifier (e.g., "STORY-001")
        project_root: Project root directory
        target_phase: Phase to reset to (must be < current phase)
        format: Output format ("text" or "json")
        workflow: Workflow type ("dev" or "qa")

    Returns:
        Exit code: 0=reset success, 1=not backward, 2=invalid ID, 3=max regressions
    """
    try:
        ps = _get_phase_state(project_root)
        state_path = ps._get_state_path(story_id)

        if not state_path.exists():
            if format == "json":
                print(json.dumps({"success": False, "error": "State file not found", "story_id": story_id}))
            else:
                print(f"State file not found for {story_id}")
            return 2

        with open(state_path) as f:
            state = json.load(f)

        current_phase = state.get("current_phase", "01")

        # Validate backward transition
        if target_phase >= current_phase:
            if format == "json":
                print(json.dumps({"success": False, "error": f"Not a backward transition: {current_phase} -> {target_phase}"}))
            else:
                print(f"ERROR: Target phase {target_phase} is not before current phase {current_phase}")
            return 1

        # Check regression limit
        regressions = state.get("regressions", [])
        if len(regressions) >= 2:
            if format == "json":
                print(json.dumps({"success": False, "error": "Maximum regressions exceeded (2)"}))
            else:
                print(f"ERROR: Maximum 2 regressions per story exceeded ({len(regressions)} already recorded)")
            return 3

        # Perform reset
        from datetime import datetime, timezone
        regression_entry = {
            "from": current_phase,
            "to": target_phase,
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "reason": "test infrastructure defect"
        }
        regressions.append(regression_entry)
        state["regressions"] = regressions
        state["current_phase"] = target_phase

        with open(state_path, 'w') as f:
            json.dump(state, f, indent=2)

        if format == "json":
            print(json.dumps({"success": True, "story_id": story_id, "from": current_phase, "to": target_phase}))
        else:
            print(f"Phase reset for {story_id}: {current_phase} -> {target_phase}")
            print(f"  Regression logged ({len(regressions)}/2 max)")

        return 0

    except ValueError as e:
        if format == "json":
            print(json.dumps({"success": False, "error": str(e), "story_id": story_id}))
        else:
            print(f"ERROR: {e}")
        return 2
```

```bash

# Exit codes:
# 0 = Phase reset successful
# 1 = Target phase >= current phase (not a backward transition)
# 2 = Invalid story ID or state file not found
# 3 = Maximum regressions exceeded (>2)
```

```markdown
### Shell Test Arithmetic Safety

When generating bash test scripts with `set -e`:

**FORBIDDEN:**
```bash
((PASSED++))   # Fails when PASSED=0: arithmetic evaluates to 0 (falsy), triggers set -e exit
((FAILED++))   # Same issue when FAILED=0
```

**CORRECT:**
```bash
PASSED=$((PASSED + 1))   # Always succeeds: assignment, not arithmetic evaluation
FAILED=$((FAILED + 1))
```

**Also remove `set -e` from test runner scripts** if the script is designed to continue after individual test failures. Use `set -uo pipefail` instead of `set -euo pipefail` when the test harness must accumulate pass/fail counts.

**Rationale:** RCA-047 — `((VAR++))` when VAR=0 evaluates the expression `0`, which is falsy in bash arithmetic. Under `set -e`, this causes immediate script exit before the increment occurs.
```
