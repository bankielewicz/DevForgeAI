# QA Command Refactoring Analysis

**Date:** 2025-11-05
**Status:** Analysis Complete - Ready for Implementation
**Token Savings Target:** 67% (from 31K to 8K character budget)

---

## Executive Summary

The `/qa` command has become "top-heavy" with 692 lines of business logic that properly belongs in the `devforgeai-qa` skill. This violates the lean orchestration principle established in Phase 3 refactoring.

**Current State:**
- Command: 692 lines, ~31,000 characters (2x budget)
- Skill: 1,331 lines, comprehensive validation logic
- Duplication: Deferral handling, report formatting, error handling exist in BOTH

**Target State:**
- Command: ~200 lines, ~8,000 characters (within budget)
- Skill: Enhanced with business logic currently in command
- Subagent: New specialized subagent for result interpretation (if needed)

## Subagent Gap Analysis: Do We Need New Subagent?

### Option A: Keep Current Architecture (No New Subagent)

**Move all Phase 2-5 logic to skill:**
- Phase 2: Result interpretation
- Phase 3: Report verification
- Phase 4: Display generation
- Phase 5: Next steps generation

**Skill grows to:** ~1,500 lines (from 1,331)
**Pro:** Single skill handles complete QA lifecycle
**Con:** Skill becomes very large, mixed responsibilities

### Option B: Create Specialized Subagent (RECOMMENDED)

**Create new subagent:** `qa-result-interpreter`

**Responsibilities:**
1. Parse QA report (read and extract structured data)
2. Determine display template based on result
3. Generate recommended next steps
4. Provide remediation guidance for failures

**When invoked:**
- In skill after generating report
- Receives: Story, QA report path, validation mode
- Returns: Structured result summary with display template and next steps

**Benefits:**
- Skill stays focused: validation logic only (~1,331 lines unchanged)
- Subagent handles interpretation (~300 lines)
- Command is pure orchestration (~200 lines)
- Clear separation of concerns
- Easier testing and maintenance

**Token efficiency:**
- Subagent in isolated context: ~8-10K tokens (haiku model)
- Skill unchanged: ~65K tokens
- Main conversation: ~8K tokens
- **Total savings:** Same tokens, but better organized

### Decision: Option B - Create `qa-result-interpreter` Subagent

**Rationale:**
1. Skill already has deferral-validator for specialized validation
2. New subagent follows established pattern (specialized task in isolated context)
3. Cleaner architecture (separation of concerns)
4. Easier to maintain and test
5. Command becomes lean orchestration (~200 lines)

## Detailed Subagent Specification

### Subagent: `qa-result-interpreter`

**Purpose:** Interpret QA validation results and generate user-facing display

**Input (from conversation context):**
- Story ID and content
- QA report path and contents
- Validation mode (light/deep)
- Story status

**Output (structured):**
```json
{
  "status": "PASSED|FAILED",
  "mode": "light|deep",
  "display": {
    "title": "✅ QA Validation PASSED",
    "summary": "...",
    "sections": [...],
    "next_steps": [...]
  },
  "remediation": {
    "violations": [...],
    "guidance": "..."
  }
}
```

**Workflow:**
1. Read QA report from file path
2. Parse report sections
3. Determine display template based on:
   - Mode (light/deep)
   - Result (pass/fail/partial)
   - Violation types
4. Generate remediation guidance
5. Recommend next steps
6. Return structured result

**Tools:** Read, Glob, Grep (view-only, no modifications)
